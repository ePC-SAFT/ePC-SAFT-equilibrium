# HELD2 Installed Completion Plan

## Outcome Proof

**Intent:** Turn the private HELD2 controller from a Stage-II-indeterminate installed candidate into a reproducible, physically certified private electrolyte-LLE candidate.
**Current Behavior:** The exact Khudaida artifact selects the dense reference and finds a negative Stage-I witness, then the single declared Stage-II local attempt fails at the unit-cube chart boundary; no installed candidate set or Stage-III result exists.
**Expected Outcome:** Full-budget installed Stage II produces at least two certified distinct candidates, generic Stage III computes every physical and free-energy certificate, and a repeatable private end-to-end artifact reaches an honest accepted or specifically bounded outcome.
**Target Output:** Four sequential mergeable leaves, one canonical installed-completion specification, and one immutable private end-to-end evidence artifact ready for the separately authorized Validation campaign.
**Owner:** Equilibrium owns HELD2 coordinates, controller behavior, numerical solvers, certificates, and evidence; the installed Provider owns EOS values, domains, pressure, and exact tensors.
**Interface:** Existing private installed HELD2 controller and Provider native capsule, with no new public optimizer or electrolyte selector.
**Cutover:** Correct the existing private path in place; remove no fail-closed gate and expose no public electrolyte route.
**Replaced Path:** The one-attempt diagnostic candidate is retained as provenance but is superseded as completion evidence by the full-budget immutable artifact; no legacy runtime or compatibility alias is restored.
**Evidence:** Exact-wheel hashes, deterministic reproduction, complete Stage-II and Stage-III traces, derivative checks, analytic/manufactured oracles, physical certificates, and gated Khudaida comparison.
**Acceptance Proof:** Two fresh non-editable installations reproduce a physically accepted private result or the same explicitly bounded scientific outcome, with every resource and certificate accounted for and no predictive comparison before physical acceptance.
**Stop Criteria:** Stop on missing exact artifacts, unexplained chart excursions, failed derivative parity, incomplete resource accounting, unavailable free-energy-gap evidence, Provider contract conflict, or any need for new authority.
**Avoid:** Do not relax tolerances, clamp meaningful bound violations, copy Provider equations, hard-code Khudaida, treat Ipopt as global, hide failures behind penalties, mutate Validation, or claim capability admission.
**Risk:** The exact Provider topology may not yield a certifiable two-liquid candidate even after numerical defects are removed; that outcome must remain a model/topology finding rather than be tuned away.

## Implementation Boundaries

**Files To Create:** Focused diagnostics or immutable evidence files only when a leaf proves they are durable scientific contracts.
**Files To Modify:** HELD2 Stage-II/III owners, private bindings, compact scientific tests, canonical architecture/status documentation, and exact candidate capture tooling.
**Files To Avoid:** Provider source, parameter catalogs, Validation repository files, promotion receipts, public electrolyte dispatch, regression code, and unrelated equilibrium routes.
**Source Of Truth:** The installed-completion specification, Perdomo formulation, exact installed Provider capsule, and live GitHub parent/leaf relationships.
**Read Path:** Frozen feed and artifact hashes enter the private controller, which evaluates Provider-owned thermodynamics through pressure roots, Stage I, Stage II, and Stage III in order.
**Write Path:** Each stage writes immutable structured diagnostics; only a fully certified Stage III result unlocks private predictive comparison and downstream Validation readiness.
**Integration Points:** Provider native SDK, deterministic pressure envelope, NLopt DIRECT-L, HiGHS Problem (64), Ipopt Problem (65), generic Ipopt Problem (67), capture script, and GitHub issue dependencies.
**Migration Or Cutover:** Land each leaf sequentially under parent issue #20, bind the next leaf to the prior merged subject, and unblock issue #23 only after the final private artifact is complete.
**Replaced Path Handling:** Preserve the issue-#22 artifact as historical provenance, replace its completion role with the new full-budget artifact, and add no duplicate controller or baseline.
**Acceptance Proof Gate:** Every leaf must pass its focused manufactured tests, installed-artifact test, full package suite, artifact/import audit, cleanup audit, and Project Truss closeout before its dependent leaf starts.
